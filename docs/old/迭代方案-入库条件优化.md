# 迭代方案：入库条件优化（底层 30 天拦截图谱）

本方案承接另外单独拆分出的《迭代方案-发布日期统一定义与取证.md》的成果，专门负责“最后一道截杀”。

在保证通过大模型漏斗输出的日期（或留空）高度合法后，使用 100% 精确的 Python `write_keyword.py` 严格计算差值，并过滤 30 天以上的老旧词汇。

## 核心设计一：日期因素移交脚本底层拦截

由于我们在前端 Prompt 里让大模型提取出了精准的日期（或对未知信息合理留空），判断这个日期“是否超期 30 天”这一动作，将不再交给大模型算术，而是交给毫无感情、不会犯算术错误的底层 Python 脚本。

### 1. 具体修改内容（`write_keyword.py`）
在接收 `args.release_date` 的逻辑附近加入拦截代码。取当前系统日期（北京时间）与大模型提供的日期计算差值：

```python
    # === 发布日期与 30 天拦截逻辑 ===
    if args.release_date:
        import datetime, timezone, timedelta
        tz_bj = timezone(timedelta(hours=8))
        today_bj = datetime.datetime.now(tz_bj)
        
        try:
            dt = datetime.datetime.strptime(args.release_date, '%Y-%m-%d')
            # 使用提取的日期强行当作当天的 0 点进行时间差计算
            dt = dt.replace(tzinfo=tz_bj)
            days_diff = (today_bj - dt).days
            
            # 若超过 30 天，属于老旧炒作词汇，直接截断
            if days_diff > 30:
                out({'ok': True, 'action': 'skip_by_date', 'msg': f'发布时间({args.release_date})距今已 {days_diff} 天，超过 30 天阈值不予入库', 'keyword': args.keyword})
                sys.exit(0) # 安全退出，不报错，不中断循环
                
            fields['发布日期'] = int(dt.timestamp() * 1000)
        except ValueError:
            out({'ok': False, 'error': f'release-date 格式错误，应为 YYYY-MM-DD: {args.release_date}'})
            sys.exit(1)
```

### 2. 差值逻辑分流处理

经过了前端提示词的严格洗礼，走到这一步的日期参数只会有三种境遇：

- `< 30 天`（正常）：说明这是最近一个月内发布的爆款工具，正常放行存库并写入飞书。
- `=""`（留空处理）：这是我们经过大模型常识过滤+谷歌搜索引擎双重漏斗，仍然全网查无发布日的**极品首发预热/泄露情报**。我们必须**极力保护并予以正常放行入库**，绝不可因为它缺少时间参数而惩罚新词。
- `> 30 天`（被拦）：完全的旧产品炒作/老版本营销，通过底层的 `sys.exit(0)` 静默跳过并安全停止执行，保卫库容。
