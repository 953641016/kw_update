# 评估报告：关键词入库条件优化（结合真实数据对比版）

在深入分析了您的本地 `~/.openclaw/workspace/memory/seo-radar-state.json` 以及现有的底层 `RULES.md` 和 `jobs.json` 抓取规则后，针对您提出的**“现有发布日期界定是否合理”**以及**“新增条件是否有漏洞”**，我出具了以下综合查勘报告。

---

## 零、 核心会诊：现行规则对「发布日期」的界定合理吗？

**结论：极不合理，标准极其混乱，是导致老旧废词泛滥的万恶之源。**

通过梳理全库 11 个任务的 Prompt（`jobs.json`），系统对 `--release-date` 的取值标准处于完全割裂的状态，具体表现为两大致命漏洞：

### 1. **“讨论日期”被张冠李戴为“产品发布日期”**
- **YouTube 任务**：明确规定“`publishedAt` 字段即为视频发布日期，直接用作 `--release-date`”。这意味着，如果今天有人发了一个“2026年如何使用 ChatGPT”的教程，系统会认为 ChatGPT 的发布日期是**今天**！
- **X (Twitter) / HackerNews 任务**：Prompt 中写着“推文输出中的日期前缀即为推文发布日期 → 直接用作 `--release-date`”。和上面同理，拿“网友发推讨论的日期”冒充“产品问世的日期”，这使得所有的 30 天拦截逻辑都会被“老产品的新讨论”瞬间击穿。

### 2. **“留空豁免权”被严重滥用（全库仅33%填写率）**
- **所有新闻/GitHub/推特任务**都有一条保底规则：“**无法从来源直接确认 → 留空（禁止估算或猜测）**”。
- 既然“留空”就能合法入库，而大模型又极度“懒惰”，导致当前本地库中 **已有高达 67% 的词条没填发布日期**（历史旧档 `release-date-filter-plan.md` 中也印证了这一点）。这就使得即使有了 30 天拦截代码，也形同虚设。

**🎯 破局之法（必须在后续迭代中纠正）**：
1. **严格剥离**：“内容抓取日期”（推文/视频发布日）与“主体产品诞生日期”。
2. **倒逼大模型取证**：Prompt 必须要求大模型在文中寻找 `launched`、`introduced` 等首发字眼。如果是老型号（如 V1.0），即使今天有新闻，只要没提 V2.0，就不予赋新值。

## 评估一：发布日期限制（30天内 或 无明确发布日期）

### 现状痛点（结合真实数据分析）
目前 `RULES.md` 允许收录的四种状态是：`✅ 正式可用`、`📣 已宣布未发布`、`🚀 刚发布`、`🔮 传言/泄露`。
由于对**`✅ 正式可用`**这一状态**没有任何时间限制**，大模型在分析 X (Twitter) 或 AI News 时，一旦看到有人讨论经典老产品，就会把它当做有效情报抓回来。
- **本地垃圾数据举例**：
  在您的 `seo-radar-state.json` 中，目前已经混入了大量毫无“首发追爆款”价值的老旧通用词，例如：
  - `ElevenLabs`
  - `Canva Magic Media`
  - `VEED AI Voice Generator`
  - `Grammarly AI expert review`
  - `vLLM`、`PaddleOCR`、`RAGFlow`

这些都是多年前或几个月前就已经火过的产品，大模型仅仅因为抓取到了相关的资讯，就将其以“状态：available”入库。这极大地浪费了飞书多维表格的空间和后续 SEO 报告生成的 API Token。

### 规则收紧后的预期疗效与漏洞排查

- **正向优化（对比现状）**：增加“发布日期 30 天内”这一硬切口后，即便推特上几万赞的推文在讨论 `ElevenLabs`，大模型也会因为该工具发布远超 30 天而将其丢弃。系统将只抓捕类似 `Wan 2.6`、`Claude 4.6` 这种真正新鲜出炉的主体。
- **豁免权（无日期/概念词）的价值**：您保留了“无日期（如泄露词、未确定词）”的豁免口子极为精准。在现有库中，`Grok 5`、`Sora 2`、`Gemini 4` 这种纯靠传言热度撑起来的词（STATUS: rumor），目前仍能被系统顺利捕获，这正是流量最大的蓝海词。

- **🚨 风险漏洞（必须防范）**：
  大模型在处理难以直接提取时间的推文/新闻时，极易使用“偷懒策略”——**直接将老词标记为“无发布日期”以绕过 30 天拦截。**
  **防范方案**：在 Prompt 中必须锁定“无日期”的定义：“仅当文本明确指示该工具【尚未发布、刚刚泄露、预告阶段】时才可判为无日期。如果是显然已存在的商业跑通产品（且文内未提及其最新 V2.0 版本的准确发布日），则直接按超期抛弃，绝不可用‘无日期’来逃避过滤！”。
  此外，还必须在 Prompt 头部位动态注入执行当天的系统日期（如 `[Today: 2026-03-27]`），以防基准期偏移。

## 综合评估结论

您的新规完全打在了现有系统的七寸上，对于**提升入库关键词的纯度**和**控制无效 API 消耗**有着立竿见影的效果。为了将该逻辑安全着陆，接下来的系统迭代实施方向应为：

1. **统一全源“发布日期”认定标准**：在各信息源（特别是 Twitter 和 YouTube）的 Prompt 抓取流程中，彻底废弃“视频/推文发布日即产品发布日”的荒谬规则，强制大模型必须在原文语境中提取真正的产品问世日期，找不到则强制留空（`""`）。
2. **后移“30天拦截”防线**：大模型仅负责寻找日期，计算 30 天是否超期的数学逻辑交由 Python 底层脚本执行，以此规避幻觉和偷懒过滤问题。
