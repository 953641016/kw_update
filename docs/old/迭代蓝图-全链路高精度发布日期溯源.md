# 迭代蓝图：全链路高精度“发布日期”溯源与入库拦截（最终版）

经过多轮评估、真实 API 测绘与成本推演，针对“老词霸榜、入库日期失真”的痛点，我们废弃了所有依靠大模型猜测或模糊匹配的不可靠手段，**确立了一条“宁缺毋滥、唯事实论”的四步闭环防线**。

本蓝图是指导 `seo-radar` 各模块重构的最终执行标准。

---

## 第一步：前端克制提取（11个抓取源的 Prompt 改造）

在 `jobs.json` 现有的抓取链路中，大模型从推文、视频摘要、新闻列表中提取 `release-date` 时，**实行最严苛的“断签”制度**：

1. **废绝“讨论日即发布日”**：如果一篇文章/推文只是在讨论某个产品，但文中**没有明确提及产品的最初问世日期**，禁止将推文发布的当天填入日期！
2. **强制留空交由底层审查**：对于找不到确切发布日的候选词，大模型必须无脑输出 `--release-date ""`（空字符串），绝不允许它利用自带知识库去盲猜。

---

## 第二步：中端微服务“分流鉴定”（`verify_date.py` 强一致性校验）

在收集到所有候选词（无论是填了日期的还是留空的）后，底层脚本接管所有留空词汇。

> ⚠️ **关键架构声明：非并发的阶梯式漏斗 (Sequential Waterfall)**
> 这三个查证渠道**绝对不是同时/并发执行的**！而是严格的“只要上一关拦截成功，立刻熔断停止后续触发”的省钱护航机制。

### 🛑 第一关：ProductHunt 官方机器直连 (准确度：100% / 成本：$0)
使用 ProductHunt GraphQL API 精确匹配 `name: keyword`。
- **命中**：拿到 `createdAt` 时间戳作为绝对发布日期，中断退出。
- **未命中/非精确匹配**：进入关卡二。

### 🛑 前置关卡二：严苛版 GitHub 开源审查 (准确度：99% / 成本：$0)
由于模糊搜索（例如搜 `Sora` 返回 `sora-test-repo`）存在致命的假阳性概率，我们强化了 GitHub 发版日期的提取规则：
- **严格匹配规则**：只有当返回仓库的 `full_name`（如 `deepseek-ai/DeepSeek-Coder`）在忽略大小写、连字符后，与搜索关键词发生**高强度的实质包含词素重合**，或仓库拥有极高 Stars (证明是正牌) 时，才采信其 `created_at`。
- **未命中或存疑**：坚决抛弃，进入关卡三。
- ***(注：原方案中的 Wikipedia API 因结构解析孱弱已被剔除出防线)***

### 🎯 终极核裁：搜索引擎 + GPT-4o-mini (准确度：极高 / 成本：单次不足 1 厘钱)
前两关找不到时，证明它必定是个闭源工具、未打榜新品，或是冷门模型。
- **动作**：通过 `s.jina.ai` 发起 `"{Keyword} official release date announcement"` 搜索。这会以 Markdown 格式瞬间抓回谷歌前十条无广告纯文字摘要。
- **降本截断（回答您的提问：喂给模型多少数据？）**：代码在提取时，会执行硬截断操作 `snippets[:1500]`，**只提取前 1500 个字符的数据**交给大模型。
  - **为何是 1500 字符？**：谷歌的排版机制决定了，针对发版日期的权威回答（Featured Snippets）必然出现在前三个搜索结果中（约 1000 字符内），后面的都是无用噪音。1500 个字母换算下来正好在 **300 个 Token 左右**，既确保了绝对不漏过核心日期，又将 API 请求费用旱涝保收地死死锁在了低于 5 厘钱的极值点。
- **宣判**：将这段精悍的 1500 字文本扔给 `GPT-4o-mini`，强制要求从摘要中扣出 `YYYY-MM-DD` 字符，或者 `NULL`，不准加一字解释。

## 第三步：底线切割杀名（`write_keyword.py` 30天物理拦阻器）

此刻，经过了前端明确和中端溯源的所有候选词，都已经拥有了一个极度精确、经得起敲打的发版日期（或经历终极侦察依然是 `NULL` 的身份）。

此时进入最终记录流程：
1. **差值计算**：取系统当天（北京时间）减去解析出的 `YYYY-MM-DD`。
2. **正常写入 (`< 30 天`)**：确认是当月新鲜爆款词，正常序列化存入 `seo-radar-state.json`，推送飞书导入。
3. **冷酷拦截 (`> 30 天`)**：明确这是过气长尾词或经典产品，脚本触发 `sys.exit(0)`，控制台打出 `[拦截] 发布距今 xx 天已超期`，从源头切断后续巨量的 SEO 生成耗费。
4. **极品保护 (`= NULL`)**：在经历所有数据库和 GPT-4o-mini 都查无对证的情况下，该词属于**全网尚未发布**或处于**极端保密的泄露阶段**。这正是情报雷达极其渴求的“零日资产”，脚本对其全面绿灯放行入库，不挂日期。

## 第四步：生产级并发控制（频率限制与防封锁）

这是您前瞻性提出的一个关乎工业级爬虫生死的重点。**这三大渠道都有极高的防并发底线，一旦短时间并发过大就会被掐断（HTTP 429 Too Many Requests）**。

在将 PoC 落地为生产代码时，必须实施以下**降频防御手段**：

1. **强制挂载 `GITHUB_TOKEN` 提升天花板**：
   - 若不带 Token，GitHub 的 Search API 限制为少得可怜的 **10 次/分钟**。我们将强制要求在环境中植入任一普通的 Personal Access Token，这样天花板会提升至 **30 次/分钟**。
   - ProductHunt（GraphQL）：自带开发者 Token 认证，上限大约能承载 **60 次/分钟** 的稳定检索。
2. **串行排队 + 随机抖动休眠 (Sleep Jitter)**：
   由于在阶段二我们确立了**“非并发执行，依次降级”**的方针。对于批量传来的上十个、甚至上百个关键词，绝不能搞多线程查询！我们要在验证不同词条的动作之间强行塞入 `time.sleep(random.uniform(1.5, 3.0))`。牺牲几分钟的入库时间，压死峰值流速，就能合法长久地白嫖三大 API 的强力数据。
