## 二、逐项深度审查

### 问题 1：竞争对手定义 — ✅ 方案正确，补充两个细节

方案提出五分类 🎯🏢🤝📰📚 是对的。但从实际案例看有两个边界情况需要明确：

**边界 1：竞品借势引流站**

seedance 报告中 `higgsfield.ai` 排第 4——这是一个 AI 视频竞品站，用"Seedance 2.0 Technical Preview"博客页借势引流到自己产品。它既不是模型官方，也不是纯套壳站。

建议在五分类中明确：**竞品借势截流站归入 🎯 竞争对手**，因为它在争夺同一批用户的注意力。

**边界 2：Google Play / App Store 中的非官方 App**

seedance 报告中 `play.google.com/store/apps/details?id=com.videoai.generator` 排第 6——这不是 ByteDance 官方 App，是第三方冒名 App。当前报告标注为"❌ 第三方"但没有进一步定义。

建议：此类标注为 **📰 媒体/平台（寄生型）**，并在报告模板中加一句风险提示："非官方 App，存在品牌侵权风险"。

### 问题 2：额外数据（发布日期/状态/热度/Trends） — ✅ 方案判断完全正确

方案给出的价值-误判矩阵是准确的。从实际案例验证：

- **Veo 3.1 Lite** 报告日期 2026-04-02，Google 官方博客发布日期 2026-03-31——仅隔 2 天。如果报告中有"发布状态：刚发布 (3天内)"这个标签，读者立刻理解为什么 SERP 被官方公告和新闻稿垄断——**这是新品红利窗口期**，SERP 还没稳定，现在建站正好。
- **seedance 2.0** 的 SERP 已经出现多个套壳站（seedance2.ai、seedance2.app），说明已有先行者占位，trends 趋势数据将直接影响"是否还值得进"的判断。

**建议补充一点**：trends_direction 如果是"下降"，应在报告结论中自动触发一个警告："此关键词热度正在衰退，建站 ROI 需重新评估"。

### 问题 3：搜索日期 → 报告日期 — ✅ 确认，纯模板改动

三份报告均使用"搜索日期"，改为"报告日期"是正确的。

### 问题 4：Semrush 查询范围 — ✅ 方案正确，但从案例看有执行落差

三份报告的 Semrush 数据现状：

| 报告 | Top10 中有 Semrush 数据的站点 | 实际竞争对手数 | 问题 |
|------|---------------------------|-------------|------|
| seedance 2.0 | seedance2.ai (AS=28)、easemate.ai (AS=54)、seedance2.app (AS=5) | 3 个套壳站 | ✅ 刚好查了竞争对手 |
| PixVerse V6 | pixverse.ai (AS=56) | 0 个第三方竞争对手 | ❌ 查的是模型官方，不应查 |
| Veo 3.1 Lite | blog.google (AS=89)、gemini.google (AS=72) | 0 个第三方竞争对手 | ❌ 全是 Google 官方 |

**关键发现**：PixVerse V6 和 Veo 3.1 Lite 的 SERP 中**根本没有第三方竞争对手**（全是官方+媒体+平台），按方案逻辑 Semrush 应该一个都不查。

这反而该触发一个 SEO 战略信号：**SERP 中无竞争对手 = 蓝海窗口**。建议在报告模板中增加判断逻辑：

```
如果 Top10 中 🎯竞争对手 = 0：
  → 报告自动输出：⚡ 蓝海信号：当前无第三方独立站竞争，建站窗口开放
```

### 问题 5：额外数据是否造成误判 — ✅ 方案结论正确

不会误判，前提是正确使用。方案提出的"关键词情报面板"设计合理。

补充一个潜在误判场景：**媒体热度高 ≠ 搜索量高**。一条 1000+ ❤ 的推文不代表用户会去 Google 搜这个词。所以面板中 Trends 数据的权重应高于媒体热度。建议在面板中排列顺序调整为：
1. Trends 7天/1月均分 + 趋势方向（最重要）
2. 发布日期 + 发布状态
3. 媒体热度（辅助参考）

## 三、方案遗漏 / 需要补强的点

### 3.1 缺少「官方 ToC 影响力」的量化框架

方案提到"模型官方是否有 ToC 业务需要区分"，但没给出判断标准。从案例看：

| 关键词 | 模型官方有 ToC？ | 对独立站的影响 |
|-------|---------------|-------------|
| seedance 2.0 | ✅ 有（seed.bytedance.com 可直接试用） | 官方占 Top1 但转化漏斗弱，独立站仍有机会 |
| PixVerse V6 | ✅ 有（app.pixverse.ai 直接创作） | 官方占 Top1-3 强壁垒，独立站应走侧翼 |
| Veo 3.1 Lite | ✅ 有（AI Studio 可试用，但有登录墙） | 官方占半壁但定价不透明，独立站打信息差 |

建议在 Stage 2 prompt 中增加一个结构化判断：

```
### 模型官方 ToC 评估
- 官方是否有面向终端用户的直接使用入口？[是/否]
- 官方 ToC 入口的转化漏斗完整度？[完整/部分/几乎无]
- 官方 ToC 对独立站的挤压程度？[强/中/弱]
- 建议应对策略：[正面竞争 / 侧翼差异化 / 信息差打法]
```

### 3.2 报告第 5 节「深度解构」在无竞争对手时应如何处理

PixVerse V6 和 Veo 3.1 Lite 的报告中，第 5 节解构的是**模型官方站**（pixverse.ai、blog.google）。按方案要求这些不应解构。

但如果完全不写第 5 节，报告会显得残缺。建议：

**当 🎯竞争对手 = 0 时，第 5 节改为「SERP 生态分析」**：
- 分析为什么没有竞争对手（太新？太小众？官方壁垒太高？）
- 分析 SERP 中哪些位置最容易被新站替代（通常是新闻稿和社交帖子的位置）
- 给出新站应切入的具体排名区间目标（如"目标 Top 5-8，替代当前的新闻/Reddit 位置"）

### 3.3 keyword_intel 的数据来源链路未说清

方案设计了 `keyword_intel` 参数，但没说清数据从哪来：

- 当 SERP pipeline 由 cron 自动触发时（`auto_serp_scan.py`），keyword_intel 应从 `seo-radar-state.json` 自动读取
- 当手动触发时（`python main_v2.py --keyword "xxx"`），需要支持 `--release-date --trends-7d` 等命令行参数

建议在方案中明确这两条链路。

### 3.4 飞书归档表缺少「竞争关系」列

方案第 10 节飞书归档表的列定义中写了要新增"竞争关系"列，但具体表格示例没有体现。需要在 Stage 3 prompt 中明确加入。
