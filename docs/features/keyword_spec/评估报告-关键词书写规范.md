# 评估报告：关键词书写规范与标准化

基于对 `openclaw` 系统核心模块（`write_keyword.py`、`dedup_check.py`、`state_io.py` 及历史数据 `seo-radar-state.json`）的深度技术与 SEO 影响分析，针对需求文档《关键词书写规范需求》中提到的“同一个关键词不同写法导致 Trends 和后续分析受影响”这一问题，现输出详细评估报告。

---

## 零、 核心结论

**现状极其严峻：关键词命名的随意性正在严重击穿系统的去重防线并造成数据失真。**
当前系统缺乏“名称规范化校正”环节，仅依赖开源社区和个人的随意写法盲目入库。这在 SEO 效果和工程架构上都留下了致命漏洞。

---

## 一、 SEO 维度评估：流量稀释与热度误判

由于大模型在提取时秉承“所见即所得”，导致生成的词汇极其混杂（如带连字符 `Luma-Uni-1`、连写 `Wan2.2`、带 V 前缀 `Suno v5`、内部小数点 `dots.ocr-1.5`等）。

### 1. Trends 查询的蝴蝶效应（致命痛点）
Google Trends 对非规范特型字符（连字号、贴合数字、v字头）极为敏感。
- **真实案例数据**：本地库中 `Wan2.2` 的 Trends 分数为 `43`，而如果是官方写法 `Wan 2.2`，热度可能有本质的区别；又比如 `Qwen3.5-Omni` Trends 获取为 `0/0`，极可能是由于连字符导致检索词断裂无法匹配真实搜索量。
- **后果**：错误的 Trends 分数直接导致真实的高热度词因为“格式不对”而被降级为绿灯（🟢低优）甚至抛弃，丧失了首发建站抢占排名的核心优势。

### 2. 页面权重内耗（Keyword Cannibalization）
如果未对书写进行官方权威性校正，同一产品（如 `Midjourney V8` / `Midjourney 8`）会生成多条记录。
未来如果自动化建站（SEO雷达优化路线图中提到了全栈发版），会针对这几个变体分别建立不同的工具页面。这在 SEO 上会造成严重的“关键词自我竞争”，分散外部链接与页面权重，最终导致任何一个页面都无法冲进 Google TOP 10。

---

## 二、 工程维度评估：去重机制失效与状态撕裂

在技术实现与代码逻辑上，目前的架构存在严重的前后不一致和认知断层：

### 1. 拦截防线的断层：`dedup_check.py` vs `state_io.py`
系统中存在两套完全不同的 normalize 规则，这是导致大量“假去重”的基础：
- **第一套（防线）**：`dedup_check.py` 中的 `normalize(kw)`。这套规则相当智能，能剔除版本号末尾0，去除 v/V 前缀等。
- **第二套（底座）**：`state_io.py` 中的 `normalize_keyword(kw)`。这套标准定义为“全局唯一入口”，并在规则注释中严令遵守，但**它仅仅做了 `.lower().strip()`**！
- **真实现状（查勘得出）**：当前在 `seo-radar-state.json` 中，`keyword_records` 的 key（也就是唯一索引）和内部的 `keyword` 字段之间，**出现了多达 511 处的差异**。这说明系统底层对于“一个词究竟该叫什么”处在混乱状态。

### 2. LLM 的“随波逐流”（根源）
从 `jobs.json` (如 X/Twitter 的 prompt) 以及 `RULES.md` 的内容来看：
虽然要求大模型去提取主角，但是遇到用户打字随意（比如推文写了 `GPT-5.4`、`dots.ocr-1.5`、`GLM-5 Turbo`），大模型就直接采用该字符串返回。
原规则中虽有一条：“*若原始名含连字符/下划线，先用 web_search 确认用户实际搜索词*”，但这完全依靠大模型自我裁量，在实际的高并发多任务中难以被严格执行。

## 三、 标准化校正环节（解决方案思路）

要从根本上解决这个问题，"**引入一个关键词书写规范校正环节，优先遵循官方写法**"是极其必要且完全正确的。

> ⚠️ **关键认知修正**：不能用硬编码正则去强制"删连字符、加空格"。
> 真实案例：`MAI Voice 1`（LLM随意提取）vs `MAI-Voice-1`（官方写法），二者在 Google Trends 和搜索结果上差异巨大。横杠是官方写法的一部分，删了反而查不到 Trends。
> 因此，解决方案的核心不是"格式化清洗"，而是**官方写法查证确权**。

初步的战术规划建议：

1. **官方写法查证确权（搭载于发布日期获取环节，零额外成本）**：
   现有的 `release_date_fetch.py` 已经在用 Sonar Pro 联网查询产品信息。只需在同一次 API 调用的 Prompt 中多要求返回 `official_name` 字段，即可零成本完成官方写法查证。
   - 对于 arXiv/HuggingFace/GitHub 等 URL 直接命中的场景，从已抓取页面标题/Model Card 中提取官方名称，完全零成本。
   - 查证后的官方名称用于 Trends 查询和飞书写入，确保与搜索引擎中的真实搜索词一致。

2. **统一底层归一化引擎（重构 state_io）**：
   将 `dedup_check.py` 中成熟的归一化逻辑迁移至 `state_io.py` 的 `normalize_keyword`，统一全局 Key 比对标准。归一化仅用于底层去重比对，不影响展示名（展示名由官方查证决定）。

3. **历史脏数据洗盘 + 飞书同步改名**：
   配备一次性洗盘脚本，对存量词重建 Key 索引、合并重复项，并对高优词批量查证官方写法，通过 Feishu API 同步更新飞书表格中的旧名称。
