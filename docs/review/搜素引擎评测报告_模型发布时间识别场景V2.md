# 搜索引擎评测报告：模型发布时间识别场景

> **日期**：2026-04-09 | **生成模型**：Claude Opus 4.6

## 一、评测背景

### 1.1 应用场景

openclaw SEO 雷达的 `verify_release_date.py` 模块负责自动获取 AI 产品的发布日期。当前采用三层瀑布流架构：GitHub Release -> Sonar Pro -> Brave Search + GPT-4o。本次评测覆盖 **6 种方案**：Serper、Brave、Tavily、Exa 四大搜索 API（配合 GPT-4o 提取）以及 Sonar、Sonar Pro 两款联网 LLM（一步到位）。

### 1.2 对搜索结果的要求

| 要求 | 说明 |
| :--- | :--- |
| **日期元数据** | API 返回结果中是否携带 `date` / `page_age` / `publishedDate` 等结构化时间字段 |
| **日期信息密度** | 搜索片段（snippet）中是否包含足够的日期上下文供 LLM 提取 |
| **精准匹配** | 搜索结果必须指向目标产品本身，而非同名的无关项目 |
| **一致性** | 多个方案提取的日期是否一致（交叉验证可信度） |
| **保守性** | 在不确定时应返回 NULL 而非幻觉一个日期 |

### 1.3 评测对象

| 方案 | 架构 | 搜索引擎 | 需要额外 LLM？ |
| :--- | :--- | :--- | :--- |
| **Serper + GPT-4o** | 两步：搜索 → LLM 推理 | Google | 是 |
| **Brave + GPT-4o** | 两步：搜索 → LLM 推理 | Brave 自有索引 | 是 |
| **Tavily + GPT-4o** | 两步：搜索 → LLM 推理 | 多源聚合 | 是 |
| **Exa + GPT-4o** | 两步：语义搜索 → LLM 推理 | Exa 神经索引 | 是 |
| **Sonar** | 一步：搜索+推理融合 | Perplexity 自有索引 | **否** |
| **Sonar Pro** | 一步：搜索+推理融合 | Perplexity 自有索引 | **否** |

### 1.5 评测方法

每个关键词通过 4 个搜索 API 各获取 5 条结果，统一由 GPT-4o 从搜索片段和日期元数据中提取发布日期。Sonar / Sonar Pro 直接以 prompt 请求提取。评测指标：

- **日期提取成功率**：10 个关键词中成功提取出日期的比例
- **误判/幻觉次数**：提取出的日期明显错误（如 1999 年、与真实值差距 > 6 个月）
- **有效提取率**：成功率减去误判率
- **日期元数据覆盖率**：API 返回的 50 条结果中有多少自带结构化日期字段

## 二、评测结论（总结先行）

> [!IMPORTANT]
> **Exa 以 60% 的日期提取成功率和 82% 的元数据覆盖率成为日期识别场景的最佳搜索 API。**
> **更关键的是：Exa 在极小众技术词（DynaVid、Omni123）上是唯一成功提取日期的 API——其他五种方案全部返回 NULL。**
> **Sonar 系列在"保守性"上远优于搜索 API——零幻觉、零误判。**

| 维度 | Serper+GPT-4o | Brave+GPT-4o | Tavily+GPT-4o | Exa+GPT-4o | Sonar | Sonar Pro |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **日期提取成功率** | 3/10 (30%) | 5/10 (50%) | 5/10 (50%) | **6/10 (60%)** | 2/10 (20%) | 2/10 (20%) |

| **有效提取率**（扣除误判） | 2/10 (20%) | 4/10 (40%) | 3/10 (30%) | **5/10 (50%)** | 2/10 (20%) | 2/10 (20%) |
| 日期元数据总量（50 条结果中） | 32 条 | 23 条 | 0 条 | **41 条** | N/A | N/A |
| 日期元数据覆盖率 | 64% | 46% | 0% | **82%** | N/A | N/A |

> [!TIP]

### 最终选型建议

| 定位 | 方案 | 理由 |
| :--- | :--- | :--- |
| **瀑布流第 ② 层（守门员）** | **Sonar** | 零幻觉、自带搜索引擎、保守策略安全，做"守门员"精准否定不存在的产品 |
| **瀑布流第 ③ 层（元数据初筛）** | **Exa publishedDate + Brave page_age** | 两者均为 ISO 8601 时间戳，可直接代码解析。Exa 元数据覆盖率最高（82%），Brave 在非学术词上更稳定 |
| **瀑布流第 ④ 层（GPT-4o 兜底）** | **Serper snippet + GPT-4o** | 仅在 ③ 无法达成一致时介入，大幅降低 LLM 调用频率 |
| **不推荐** | Tavily | 零元数据、高幻觉率（2/5 提取结果为误判） |

## 四、分类对比分析

### 4.1 架构对比

| 维度 | Serper/Brave/Tavily + GPT-4o | Exa + GPT-4o | Sonar / Sonar Pro |
| :--- | :--- | :--- | :--- |
| **架构** | 两步：关键词搜索 → LLM 推理 | 两步：语义搜索 → LLM 推理 | 一步：搜索与推理融合 |
| **搜索引擎** | Google（Serper）/ Brave / 多源（Tavily） | Exa 自有神经索引 | Perplexity 自有索引 |
| **优势** | Google 覆盖面广，snippet 丰富 | arXiv/GitHub 源头追溯能力最强，元数据最丰富 | 内置 citation 验证，零幻觉 |
| **劣势** | 极小众词覆盖不足 | 消费级产品词不如 Google 覆盖广 | 过于保守，对模糊信号选择放弃 |
| **适合场景** | 消费级产品词日期识别 | 学术/技术类产品日期识别 | 高精度需求（宁可漏报也不误报） |

5 种搜索方案都成功提取日期。日期范围 2026-04-02 ~ 2026-04-04（差异仅 2 天），可信度极高。Serper 出现严重偏差（2023-10-23），可能是 snippet 中混入了旧新闻内容导致 GPT-4o 误判。

- **Exa/Tavily** 返回 04-03（新闻报道日期）
- **Sonar/Sonar Pro** 返回 04-04（可能是 GitHub 开源日期）

#### Seedance 2.0 — 三方共识，高可信度

- Serper/Tavily → 2025-08-05：Eleven Music 模型首发日
- Brave/Exa → 2026-04-01：ElevenMusic iOS App 上架日
- 两组日期都"正确"，只是指向不同的里程碑事件。Exa 的 snippet 全部来自 2026 年 4 月的新闻报道，偏向最近事件。

## 五、对 `verify_release_date.py` 的优化建议

### 5.1 推荐瀑布流架构

```
① GitHub Release (免费、精确、适合开源项目) — 不变
② Sonar (降级为普通版，省成本) — 做"守门员"，精准否定不存在的产品
③ Exa publishedDate 初筛 — 元数据覆盖率最高(82%)，ISO 8601 直接解析
④ Brave page_age 交叉验证 — ISO 时间戳直接解析，与 Exa 交叉比对
⑤ GPT-4o 仅在 ③④ 结果不一致时介入 — 大幅降低 LLM 调用频率
```

### 5.2 具体优化点

1. **Sonar Pro → Sonar 降级**：两者表现完全相同，Sonar 成本仅 1/3
2. **新增 Exa 搜索层（③）**：41/50 条结果自带 `publishedDate`，覆盖率 82%，尤其擅长学术/技术词
3. **Brave `page_age` 交叉验证（④）**：ISO 8601 格式，与 Exa 数据格式统一
4. **双源交叉验证规则**：Exa.publishedDate 和 Brave.page_age 一致（差距 < 7 天）时直接采信，无需调用 GPT-4o
5. **Tavily 排除**：零元数据 + 高幻觉率，不适合此场景

### 5.3 预期收益

| 指标 | 当前架构 | 优化后 |
| :--- | :--- | :--- |
| GPT-4o 调用频率 | 每次兜底都调用 | 仅在结构化日期冲突时调用（预计减少 70%） |
| Sonar 成本 | $0.003/1K (Pro) | $0.001/1K (普通版) |
| 日期提取覆盖率 | Sonar 20% + Brave 50% | Exa 60% + Brave 50%（并集约 70%） |
| 极小众词覆盖 | 基本为 0% | Exa 独占成功 2/2（DynaVid、Omni123） |
| 误判率 | 依赖单源判断 | 双源交叉验证，误判率大幅降低 |
