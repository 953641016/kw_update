## 背景

SEO 雷达从 11 个信息源每天抓取候选关键词，但并非所有词都值得建站。需要一套清晰、可执行的评估体系，帮助 OpenClaw agent 在入库时自动判定优先级，让编辑团队聚焦高价值词。

### 核心意图（摘自项目说明）

> 找到关键词 → 快速上新站 → AI SaaS 站，冲 Google 搜索 TOP10

一切评估逻辑都服务于这个目标：**越早发现、越快建站、越能带来付费用户的词，优先级越高。**

## 一、四维定义

### 1. 新：发布距今 ≤ 7 天

**问的问题：** 这个词是不是刚出来的？

**判断方式：** 纯算术，基于 `release-date` 自动计算，无需大模型判断。

```
距今天数 = 当前UTC日期 - 发布日期
≤ 7天 → ✅ 满足「新」
> 7天 → ❌ 不满足
无日期 → ❌ 不满足（但不影响入库，见入库条件优化方案）
```

**为什么重要：** SEO 抢先占位的时间窗口极短。词越新，SERP 竞争越小，越有机会围绕该关键词快速上新站，冲 Google 搜索 TOP10。

> 📝 7 天是「首发抢占窗口」的定义，不覆盖全部流量爬升期。Google Trends 搜索量爬升通常滞后 2-4 周，但「新」维度关注的是 SERP 空位期，不是流量峰值期。

### 2. 真：确实存在，有可信的一手证据

**问的问题：** 这个东西是真的吗？有据可查吗？

**本质：** 风控过滤器。防止为不存在的东西建站，白费精力。

**「真」不等于「现在能用」。** 官方已宣布但未发布、可信来源的泄露信息，只要有一手证据，都算「真」。

#### ✅ 算「真」的情况

**不需要额外查资料。** 各来源任务本身就自带证据——从 GitHub 抓到的词天然有 repo，从 HuggingFace 抓到的天然有 model card，从 X/ProductHunt 抓到的天然有帖子/产品页。「真」的判断基于来源任务已获取的上下文信息即可。

| 来源任务 | 天然具备的证据 |
|---------|---------------|
| seo-github | 有 GitHub repo（代码/权重已开源） |
| seo-huggingface | 有 HuggingFace model card |
| seo-producthunt | 有 ProductHunt 产品页 |
| seo-twitter | 有推文，看发布者是否为官方账号或可信大 V |
| seo-ainews | 有科技媒体报道 |
| seo-youtube | 有演示视频 |
| seo-hackernews | 有 HN 讨论帖（通常附带 Demo 链接） |
| seo-arxiv | 有论文，需检查是否附带代码/权重链接 |
| seo-platforms | 有推理平台模型页（fal.ai / wavespeed / replicate / openrouter 上架） |
| seo-china | 有官方站模型列表 或 量子位等媒体报道 |

#### ❌ 不算「真」的情况

| 情况 | 为什么过滤 |
|------|-----------|
| 只有一篇 arXiv 论文提了个方法名，无代码无权重无 Demo | 纯学术概念，可能永远不会有搜索量 |
| 只有一个营销号/个人博客在说，无法交叉验证 | 蹭流量造词，不可信 |
| 只有"我们计划做 XXX"的一句话，无产品截图/Demo/时间线 | 纯画饼，吹牛 |

#### 与发布状态的关系

| 发布状态 | 默认是否「真」 | 说明 |
|---------|:------------:|------|
| ✅ 正式可用 | ✅ | 有 API/Demo，铁板钉钉 |
| 🚀 刚发布(<7天) | ✅ | 同上 |
| 📣 已宣布未发布 | ✅ | 官方背书，产品形态明确 |
| 🔮 传言泄露（可信来源） | ✅ | 多方交叉验证 |
| 🔮 传言泄露（来源不明） | ❌ | 单一来源，无法验证 |
| 纯论文方法名 | ❌ | 只有概念，无产品化迹象 |

### 3. 热：在多个平台被讨论

**问的问题：** 这个词有没有引起广泛关注？

**判断方式：** `keyword_media_heat` 中出现 **≥2 个不同平台前缀** 即满足。不区分供给侧与第三方——官方推文自带流量光环，计入热度符合实际情况。

#### ✅ 算「热」的组合举例

- X (Twitter) + Hacker News 同时讨论
- YouTube 有演示视频 + Reddit 有讨论帖
- TechCrunch 报道 + ProductHunt 上架
- arXiv 发论文 + GitHub 有开源 repo
- 官方 X 发布 + HuggingFace 上传权重

#### ❌ 不算「热」

- 只在一个平台有一条帖子
- 转发/搬运同一条消息不算独立来源

#### 工程实现：State 文件交叉查

各 seo-* cron 任务是隔离执行的，单个任务无法看到其他任务的数据。「热」的判断通过 **State 文件交叉查** 实现：

**已验证的数据格式**：`seo-radar-state.json` 中 `keyword_media_heat[keyword]` 存储的是多行文本，每行以平台 emoji 前缀开头（`🐦 X:` / `🔺 PH:` / `🐱 GH:` / `👽 Reddit:` / `📰 News:` / `📺 YT:` / `🤗 HF:` / `🟠 HN:`）。`write_keyword.py` 的 `merge_media_heat()` 函数用 `prefix = new_line.split(':')[0]` 做平台去重。**此格式已稳定运行，可直接用于平台计数。**

1. **入库时判断**：当某个 seo-* 任务准备写入一个词时，先检查 `seo-radar-state.json` 中该词的 `keyword_media_heat` 字段，统计已有几个**不同平台前缀**的热度记录。若「已有记录的平台数 + 本次任务平台 ≥ 2」→ ✅ 满足「热」。
2. **首次入库**：词首次入库时，只有当前任务一个来源，「热」通常为 ❌（除非当前任务的上下文中明确提到多个独立平台的讨论）。
3. **优先级重评**：由 `priority_reeval.py` 独立脚本执行（由 `release-status-update` cron 每日触发），检查所有 🟡中 优先级的词，若 `keyword_media_heat` 已积累 ≥2 个不同平台前缀 → 自动升级为 🔴高。

### 4. 赚钱：搜这个词的人想找工具/产品

**问的问题：** 搜索这个词的人，是潜在的付费用户吗？

**本质回归：** 变现路径是 `关键词 → 围绕关键词上新站 → 冲 Google TOP10 → 流量 → 转化`。「赚钱」评估的不是"这个产品能不能挣钱"，而是**"搜这个词的人会不会变成我们新站的付费用户"**。

核心判断：**搜索意图是否指向「找工具 / 了解产品 / 对比选择」**。

#### ✅ 算「赚钱」

| 信号 | 说明 | 举例 |
|------|------|------|
| **词是产品名/功能名** | 用户搜它是想了解、试用或对比这个产品 | Sora、Veo 3、Kling Elements |
| **存在衍生搜索意图** | 有 "X alternatives"、"X vs Y"、"how to use X"、"X pricing" 等搜索派生 | "Runway alternatives" 说明用户在选工具 |
| **SERP 中有工具型结果** | 搜索结果出现了 AI SaaS 站、工具评测页 | 说明 Google 认为这是工具意图 |
| **同品类已有商业产品** | 该品类不是空中楼阁，有成熟的付费用户群 | 视频生成赛道有 Runway/Pika/Kling 在变现 |

#### ❌ 不算「赚钱」

| 情况 | 说明 | 举例 |
|------|------|------|
| **纯技术概念/算法名** | 搜索者只想学习原理，不会付费 | "Diffusion Model"、"FlowMatching" |
| **导航型搜索** | 用户只是找官网入口，不会停留在第三方站 | "ChatGPT login"、"Midjourney sign up" |
| **SERP 全是论文/教程/百科** | Google 判定这是学习意图，工具页无法获得排名 | 搜索结果全是 arxiv / medium / wikipedia |
| **过于宽泛的品类词** | 竞争极高，新站无法排名 | "AI"、"machine learning" |

## 二、「真」与「赚钱」的边界澄清

这两个维度容易混淆，必须明确分工：

| 维度 | 视角 | 问的问题 | 过滤什么 |
|------|------|---------|---------|
| **真** | 供给侧 | 这个产品/工具是真实的吗？ | 学术空气词、蹭流量造词、纯画饼 |
| **赚钱** | 需求侧 | 搜这个词的用户有付费意图吗？ | 纯学习意图词、导航词、无转化价值词 |

**不重叠：**
- 一个词可以「真」但不「赚钱」：开源推理框架 vLLM，真实存在但搜它的人不会在 jxp.com 付费
- 一个词可以「赚钱」但待验「真」：泄露的某大厂新产品名，搜索意图指向工具，但产品还没实锤

## 三、定级规则

| 优先级 | 条件 | 含义 |
|:------:|------|------|
| **🔴 高 (high)** | 满足 **2 个及以上** 维度 | 高价值词，优先建站 |
| **🟡 中 (mid)** | 满足 **1 个** 维度 | 有一定价值，正常排队 |
| **🟢 低 (low)** | **都不满足**，但符合基础收录标准 | 占位型收录 |

### ⚠️ 优先级上限约束

以下关键词即使满足 ≥2 个维度，**优先级上限为 🟡中**：

| 约束条件 | 原因 |
|---------|------|
| `keyword-type = technical`（🔬 技术词） | 搜索者以学习为主，转化率远低于产品词。学术词天然满足「新」+「真」，不设上限会系统性高估 |
| 工具品类 = Text（文本类） | jxp.com 核心定位是 Video/Image/Audio 工具站，Text 品类竞争激烈（ChatGPT/Claude 等巨头垄断），新站难以获得排名和转化 |
| 工具品类 = Agent（智能体类） | Agent 赛道目前以企业级/开发者为主，C 端付费意愿低，且品类定义模糊，建站 ROI 不确定 |

> 仅 `keyword-type = product/demand` **且** 工具品类为 Video/Image/Audio 的词可获得 🔴高 优先级。

### 典型场景举例

| 关键词 | 类型 | 品类 | 新 | 真 | 热 | 赚钱 | 定级 |
|--------|:----:|:----:|:--:|:--:|:--:|:----:|:----:|
| Veo 3.1（Google刚发布3天） | product | Video | ✅ | ✅ | ✅ | ✅ | 🔴高 |
| 某大厂泄露的新视频模型（多方报道） | product | Video | ✅ | ✅ | ✅ | ✅ | 🔴高 |
| Sora（已上线半年） | product | Video | ❌ | ✅ | ❌ | ✅ | 🟡中 |
| FlowMotion（arXiv论文，有代码） | technical | Video | ✅ | ✅ | ❌ | ❌ | 🟡中（技术词上限） |
| Claude 4（刚发布） | product | Text | ✅ | ✅ | ✅ | ✅ | 🟡中（Text 品类上限） |
| Manus Agent（刚发布） | product | Agent | ✅ | ✅ | ✅ | ✅ | 🟡中（Agent 品类上限） |
| DiffusionBridge（纯论文，无代码） | technical | Image | ✅ | ❌ | ❌ | ❌ | 🟢低 |
| 某营销号自创的 "AIVideoMax" | — | — | ❌ | ❌ | ❌ | ❌ | ❌ 不收录 |

### 学术新词特例（保留现有规则）

arXiv / HuggingFace 来源 + Trends=0 的词，不因缺少热度数据而降级。使用现有学术词收录标准（机构背景、对标模型、SERP 空位、功能明确）替代 Trends 分数评估。满足 2 条及以上 → 🟢低 收录，SEO 分析注明「预占型」。

### cron prompt 清理说明

多个 seo-* cron prompt（`jobs.json`）中硬编码了旧的优先级排序规则（基于 Trends 分数的 🔴高/🟡中/🟢低 定义）。更新 RULES.md 后，需**清理 prompt 中与优先级判断相关的内嵌规则**，只保留对 RULES.md 的引用（`cat /root/.openclaw/workspace/scripts/RULES.md`），避免 LLM 面对 prompt 与 RULES.md 的矛盾指令。

清理时注意：**保留** `cat /root/.openclaw/workspace/scripts/RULES.md` 这行指令（让 LLM 知道去哪里读规则），**删除** prompt 里直接内嵌的优先级排序逻辑（如 「Trends分数7天值>20 → 🔴高」等）。

### priority_reeval.py（新增独立脚本）

**为什么独立**：`release_status_update.py` 现有职责是发布状态更新（🚀→✅），是幂等的。如果在其中内嵌优先级重评逻辑，多个 seo-* 任务同天更新同一个词的热度时，可能触发重复升级写入。独立脚本可做幂等保护，且便于单独触发库存重评。

**功能规格**：

```
python3 priority_reeval.py [--dry-run] [--full]
```

- **默认模式**（无 `--full`）：仅检查 🟡中 优先级的词
- **全量模式**（`--full`）：对所有词按四维规则重新评估（用于库存重评验证，见第六节）
- **`--dry-run`**：仅打印，不写飞书

**重评逻辑**：

1. 拉取飞书全量记录（复用 `feishu_fetch_all.py` 缓存）
2. 读取 `seo-radar-state.json` 获取每个词的 `keyword_media_heat`
3. 对每个目标词，计算四维：
   - **新**：`发布日期` 距今 ≤ 7 天
   - **真**：默认 ✅（已入库的词一定通过了 RULES.md 的收录标准）
   - **热**：`keyword_media_heat` 中 ≥2 个不同平台前缀
   - **赚钱**：`keyword-type = product/demand` → ✅；`technical` → ❌
4. 按定级规则计算应有优先级（含 technical 上限 + Text/Agent 品类上限约束）
5. 若计算结果与当前优先级不同 → 加入更新队列
6. **幂等保护**：若当前优先级已是计算结果 → 跳过（避免重复写入）
7. 批量更新飞书

**触发方式**：`release_status_update.py` 在所有 Case 处理完毕、批量写入飞书后，末尾调用：

```python
import subprocess
subprocess.run([sys.executable, os.path.join(os.path.dirname(__file__), 'priority_reeval.py')],
               check=False)
```

## 五、给 OpenClaw 的判断指引

> 对每个候选词，按以下顺序逐一检查四个维度，记录满足了哪几个，然后按定级规则确定 priority：
>
> 1. **「新」检查**：release-date 距今 ≤ 7 天？→ 纯算术，脚本自动判断
> 2. **「真」检查**：来源任务的上下文中有一手证据吗？→ 基于已有来源信息判断，不额外查资料
> 3. **「热」检查**：该词在 seo-radar-state.json 的 `keyword_media_heat` 中是否已有其他平台的热度记录？统计不同平台前缀数量（🐦/🔺/🐱/👽/📰/📺/🤗/🟠）。若「已有记录的平台数 + 本次任务平台 ≥ 2」→ ✅。若该词是新词（state 中无记录），仅当本次任务上下文明确提及多个独立平台讨论时才满足。
> 4. **「赚钱」快捷判断**（短路评估，减少不必要的SERP查询）：
>    - `keyword-type = product` → **默认 ✅**（产品词天然有工具搜索意图）
>    - `keyword-type = demand` → **默认 ✅**（需求词指向功能/场景）
>    - `keyword-type = technical` → **默认 ❌**（技术词搜索者以学习为主）
>    - 若前三步已满足 ≥2 维度且 keyword-type ≠ technical → 可跳过 SERP 分析，直接定 🔴高
>    - 仅在需要区分 🟡中 vs 🔴高 时，才走 SERP 深度分析
>
> **定级公式：**
> - `keyword-type = technical` → 上限 🟡中（即使满足 ≥2 维度）
> - 工具品类 = Text 或 Agent → 上限 🟡中（即使满足 ≥2 维度）
> - 其余（product/demand 且品类为 Video/Image/Audio）：满足 ≥2 个 → `--priority high`
> - 满足 1 个 → `--priority mid`
> - 满足 0 个（但符合收录标准）→ `--priority low`

## 六、库存重评（验证方案）

新方案上线后，需要对已入库的全量关键词执行一次四维重评，目的：

1. **验证方案效果**：统计新旧规则下优先级分布的变化，确认新规则是否符合预期
2. **发现边界 case**：用真实生产数据暴露四维定义中的歧义或遗漏
3. **生成对照报告**：为后续方案调优提供数据基础

### 执行方式

```bash
python3 priority_reeval.py --full --dry-run    # 先 dry-run 看报告
python3 priority_reeval.py --full               # 确认后正式执行
```

### ⚠️ 空热度词处理

`keyword_media_heat` 是近期迭代新增的功能，早期入库的关键词没有热度数据。`--full` 模式下对这类词的处理规则：

- **热度为空的词**（`keyword_media_heat` 中无该词记录）：「热」维度视为**无法判断**（而非 ❌），**不参与自动升降级**
- 报告中单独列出这些词，标注 `[热度数据缺失]`，供人工复核
- 同时标注 `keyword_media_heat` 为空且 `keyword-type=technical` 的词（早期纯论文方法名词可能未经过「真」过滤器），供人工确认是否应保留

> 仅对有热度数据的词执行自动重评。空热度词保持原优先级不变，等后续 cron 补充热度数据后再由日常重评处理。

### 输出报告格式

```
[REEVAL REPORT] 2026-04-02
  总词数: 280
  有热度数据: 180
  重评变更: 45
    🟢低 → 🟡中: 12  (典型: FlowMotion, SkyReels V3)
    🟡中 → 🔴高: 8   (典型: Veo 3.1, Kling Elements)
    🔴高 → 🟡中: 25  (典型: Diffusion Model [技术词上限])
  未变更: 135
  [热度数据缺失] 跳过: 100  (不自动升降级，待人工复核)
    其中 technical 词: 35  (可能未经「真」过滤，建议人工确认)
```

### 时序

1. 先更新 RULES.md + 清理 cron prompt（让新入库的词使用新规则）
2. 上线 `priority_reeval.py`
3. `--full --dry-run` 生成对照报告 → 确认无误后执行
4. 此后由 `release-status-update` cron 每日触发（默认模式，仅检查 🟡中 词）
