# 双引擎趋势监控系统实施方案 (V9 - 终版)

## 一、核心原理

### 为什么需要锚点？
Google Trends 返回的 0-100 是**相对值**，单独查一个词的"50"和另一个词的"50"不能比较。
把目标词和固定锚点**放在同一次查询**中，就能得到可比较的比例关系。

### 为什么需要双锚点？
经大量实测发现：**没有任何单一锚点能同时兼容大词和小词**。
- 大锚点（如 Kling=83）：小词（如 Vidu Q3）被碾压为 0
- 小锚点（如 infinitetalk=25）：大词（如 Sora）把锚点碾压为 0

**解决方案**：一次查询发送 `[目标词, 小锚点, 大锚点]`，根据哪个锚点有数据来自适应选择。
Google Trends API 支持一次最多 5 个关键词，三词查询仅消耗 1 次 API 调用。

### 跨品类说明
双锚点均为视频赛道词。用它们衡量文本/音频类词（如 Claude、ElevenLabs），穿透率的**绝对值**跨品类不完全精确。
但本系统的核心目标是**纵向跟踪**（同一个词这周比上周热了多少），而非横向比较（Claude 和 Kling 哪个更热）。
穿透率的**趋势比**（7天÷30天）不受跨品类影响——只要锚点稳定，比值就能正确反映方向。

## 二、锚点配置（经实测验证）

全赛道共用一组双锚点（不按品类区分，简化配置）：

| 角色 | 词 | 实测值域 | 选择理由 |
|------|-----|:---:|--------|
| 小锚点 | **seedance** | 24-35 | 体量小但不归零；曲线平稳；极小词（如 Kling Motion=3）仍可见 |
| 大锚点 | **Kling** | 51-83 | AI 视频中坚品牌；体量适中稳定；不像 Sora/wan 那样剧烈波动 |

**实测验证截图数据**（2026-03-26，Past 30 days）：
```
三词联合查询 [Kling Motion, seedance, Kling]：
  Kling = 83, seedance = 35, Kling Motion = 3
  穿透率 = 3/35 = 8.6% ✅ 有效！
```

**锚点健康检查**（代码层面）：
- 每次批量执行前，先单独查询一次 `[seedance, Kling]`
- **存活检测**：如果任一锚点 7 天平均值 < 5 → 告警并暂停该轮
- **漂移检测**：如果任一锚点 Velocity > 2.0（如 Kling 发新版导致飙升）→ 日志告警 `⚠️ 锚点异常波动，本轮结果仅供参考`，继续执行不停止（避免数据断档）

**配置文件** (`anchors.json`)：
```json
{
  "small_anchor": "seedance",
  "big_anchor": "Kling"
}
```

## 三、自适应取值逻辑

每次查询发送：`[目标词, seedance, Kling]`

```
取 value[0] = 目标词值 K
取 value[1] = 小锚点值 S (seedance)
取 value[2] = 大锚点值 B (Kling)

自适应选择分母：
  如果 S > 0 且 B > 0 → ratio = K / S（用小锚点，精度更高）
  如果 S = 0 且 B > 0 → ratio = K / B（目标词很大，碾压了小锚点）
  如果 S = 0 且 B = 0 → ratio = K（目标词极大，直接记录原始分）
```

## 四、均值计算流程（完整示例）

### 第一步：拉取 30 天原始数据

对 Google Trends 发起一次三词联合查询 `[Grok Imagine, seedance, Kling]`，取回 30 天按天数据：

```
Day 01: Grok_Imagine=2,  seedance=30, Kling=82
Day 02: Grok_Imagine=3,  seedance=28, Kling=85
...
Day 26: Grok_Imagine=13, seedance=35, Kling=75
Day 27: Grok_Imagine=10, seedance=32, Kling=78
Day 28: Grok_Imagine=12, seedance=34, Kling=80
Day 29: Grok_Imagine=15, seedance=33, Kling=76
Day 30: Grok_Imagine=11, seedance=31, Kling=79
```

### 第二步：逐日算穿透率

每一天：`ratio = 目标词值 / 小锚点值`（小锚点 seedance 非零时优先使用）

```
Day 26: ratio = 13 / 35 = 0.371 (37.1%)
Day 27: ratio = 10 / 32 = 0.313 (31.3%)
Day 28: ratio = 12 / 34 = 0.353 (35.3%)
Day 29: ratio = 15 / 33 = 0.455 (45.5%)
Day 30: ratio = 11 / 31 = 0.355 (35.5%)
```

### 第三步：算均值

```
7天均值  = 最近 7 天 ratio 的平均值
         = (Day24~Day30 的 ratio 之和) / 7
         = 假设算出 0.368（即 36.8%）

30天均值 = 全部 30 天 ratio 的平均值
         = (Day01~Day30 的 ratio 之和) / 30
         = 假设算出 0.281（即 28.1%）
```

### 第四步：算爆发速度（Velocity）

```
Velocity = 最近 2 天穿透率均值 / 前 5 天穿透率均值

以上面数据为例：
  最近 2 天均值 = (Day29 + Day30) / 2 = (0.455 + 0.355) / 2 = 0.405
  前 5 天均值   = (Day24~Day28) / 5 = 假设为 0.342
  Velocity = 0.405 / 0.342 = 1.18
```

**含义**：Velocity 反映的是**短期加速度**——最近 2 天比前 5 天猛了多少倍。
- Velocity > 2.0 → 短期内突然爆发（如新产品发布）
- Velocity ≈ 1.0 → 平稳运行
- Velocity < 0.5 → 短期内急速降温

**使用场景**：
- **入库 ≤ 7 天**的新词没有 30 天数据，无法算"趋势比"，此时用 Velocity 代替判定趋势方向
- **入库 > 7 天**的词用"趋势比"（7天均值÷30天均值）为主，Velocity 作为辅助参考

### 第五步：算趋势（含爆发检测）

按优先级从高到低逐条判定：

```
【最高优先级】检测爆发：
  如果 Velocity > 3.0 → 🚀 爆发（不论有没有30天数据）
  含义：最近 2 天比前 5 天猛了 3 倍以上，属于突发事件级别

情况 A（≥ 7 天数据，且未触发爆发）：
  趋势比 = SMA_7 / SMA_30
  趋势比 > 2.0  → 🚀 爆发（持续性爆发：整周热度是月平均的 2 倍以上）
  趋势比 > 1.25 → ⬆️ 上升
  趋势比 0.75~1.25 → ➡️ 平稳
  趋势比 < 0.75 → ⬇️ 下降

情况 B（3-6 天数据）：
  简化 Velocity = 最后1天穿透率 / 前 (N-1) 天穿透率均值
  阈值同上（>3.0 爆发, >2.0 上升, 0.5~2.0 平稳, <0.5 下降）

情况 C（0-2 天数据）：
  数据点太少，任何计算都不可靠
  → Trends 趋势 = 🚫 无数据
```

### 最终写入飞书

- **Trends 值** = `13/8`（目标词原始分的 7天均值 / 30天均值，原始分天然 0-100，无超限问题）
- **Trends 趋势** = `⬆️ 上升`（基于穿透率链路的趋势比 + Velocity 判定）

**边界格式**：
```
≥ 7 天数据  → "13/8"（正常格式）
3-6 天数据  → "13/-"（30天数据未就绪，"-" 占位）
0-2 天数据  → "-/-" （数据不足，趋势同时写 🚫 无数据）
```
RULES.md 解析时统一取 `"/"` 前部分，`"-"` 视为 0。

## 五、最终产出：升级现有飞书 Trends 两列

**不新增飞书列**，升级现有两列的计算逻辑：

| 现有飞书字段 | 类型变更 | 旧算法 | 升级后算法 |
|:---:|:---:|------|------|
| **Trends 值** | Number → **Text** | 单次快照 0-100 | → 目标词原始分的 `7天均值/30天均值`（如 `13/8`） |
| **Trends 趋势** | 单选（不变） | 近7天逐日差值法 | → 穿透率 Velocity + 趋势比判定 |

**为什么存原始分而非穿透率？**
穿透率可超 100%（大词 ratio = 100/5 = 2000%），不直观。原始分天然 0-100，SEO 团队一眉能懂。
虽然原始分跨批次不完全可比，但所有词都用同一组锚点查询，标尺基本一致。
穿透率保留在内部计算链路中，仅用于趋势判定，不露出给用户。

```
同一次查询 [Grok Imagine, seedance, Kling] 返回每天三个值

Trends 值 → 取 value[0]（目标词原始分），算 7 天和 30 天平均
            示例：7天均值=13, 30天均值=8 → 写入 "13/8"

Trends 趋势 → 内部用 ratio = value[0]/value[1] 算穿透率 SMA、Velocity
              示例：趋势比=1.31 → 写入 "⬆️ 上升"
```

**趋势判定规则**（四档 + 无数据）：
```
Velocity > 3.0 或 趋势比 > 2.0 →  🚀 爆发（突发或持续性爆量）
趋势比 > 1.25                  →  ⬆️ 上升
趋势比 0.75~1.25               →  ➡️ 平稳
趋势比 < 0.75                  →  ⬇️ 下降
0-2 天数据 或 获取失败          →  🚫 无数据
3-6 天数据                     →  用简化 Velocity 替代（同阈值）
```

**RULES.md 适配**：
`Trends 值` 改为 Text 后，现有数字比较规则需更新：
```
旧：Trends > 20 → 🔴高优先
新：取 Trends 值 "/" 前部分（7天均值）做数字比较
     如 "13/8" → 取 13 → 13 < 20 → 🟡中优先
```
`write_keyword.py --trends-score` 需改为 `type=str`，接受 `"13/8"` 格式。

## 六、运行轴线

| 阶段 | 触发条件 | 调度方式 | 计算范围 |
|------|---------|---------|---------|
| 第 0 天 | 入库时 | CLI 单次调用 | 首次快照，同时登记 tracker |
| 第 1-7 天 | 入库天数 ≤ 7 且 ≥ 12h | 每日 Cron | 仅算 7天均值 |
| 第 8-30 天 | 7 < 入库天数 ≤ 30 | 周一、四 Cron | 算 7天 + 30天均值 |
| >30 天 | — | 暂不实施 | — |

**API 防抖**：
- 每次查询间隔 `random(5, 15)` 秒
- 遭遇 429 时暂停 `60-300` 秒后重试（最多 3 次）

**Cron 时间**（避开现有爬虫）：
```bash
30 08 * * * python3 anchor_trends_monitor.py --mode daily
30 09 * * 1,4 python3 anchor_trends_monitor.py --mode weekly
```

## 九、工程风险与对策

| 风险 | 说明 | 对策 |
|------|------|------|
| 批量执行耗时 | 1000 词 × 间隔 5-15s = 1.4-4 小时 | 实现断点续传（tracker 记录每轮进度）；分批执行 |
| tracker 文件膨胀 | 存全库每日穿透率 | 只保留最近 30 天滑动窗口，超期数据自动清除 |
| API 稳定性 | 无官方 API，429 限流频繁 | 已设 random sleep + 退避；严重时切换代理 IP |
| 锚点漂移 | 锚点词自身爆发导致全库穿透率假性下降 | 健康检查增加 Velocity > 2.0 告警 |
| 锚点失活 | 锚点词停运导致持续为 0 | 健康检查 7天均分 < 5 暂停 + 告警，需人工更换锚点 |
