# 迭代方案：关键词媒体热度字段

## 背景与目标

Google Trends 对新词存在 3–7 天的滞后期，无法在词刚发布时提供有效热度信号。本方案新增`媒体热度`字段，汇聚各数据源平台的热度数据，作为 Trends 的前导补充指标，辅助编辑团队在 Trends=0 时也能自信判断词的价值并提前建站卡位。

---

## 一、字段设计

### 1.1 飞书多维表格新增字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `媒体热度` | 多行文本 | 各平台热度数据的复合结构，由脚本格式化写入 |

### 1.2 各平台展示格式（标准模板）

```
🐦 X(Twitter): {n}条推文 | 最高 {max_likes}❤ | {top_rt}RT
🔺 ProductHunt: {votes} Votes ({rank})
🐱 GitHub: {stars} Stars
👽 Reddit: {n}条帖子 | 最高 {max_score} Upvotes
📰 新闻媒体: {n}篇报道 ({outlets})
更新于 YYYY-MM-DD UTC+8
```

展示示例：
```
🐦 X(Twitter): 15条推文 | 最高 12,300❤ | 2,100RT
🔺 ProductHunt: 850 Votes (#1 of the Day)
👽 Reddit: 12条帖子 | 最高 2,100 Upvotes
更新于 2026-03-27 UTC+8
```

> **关键设计决策1**：X 的 RT 数取**最高❤帖对应的RT**（非所有帖最高RT），确保❤和RT描述同一条帖子，数字有联动语义。
> **关键设计决策2**：字段末行统一记录**最后更新时间**（北京时间日期，标注 UTC+8），每次任何平台写入时均刷新此行。日期由脚本执行 `date +%Y-%m-%d` 获取（服务器已为北京时间，或通过 `TZ=Asia/Shanghai date +%Y-%m-%d` 强制）。

### 1.3 字段写入规则

- **仅写入本次抓取批次覆盖的平台**。seo-twitter 只填 X 行，seo-producthunt 只填 ProductHunt 行，不同行之间相互独立。
- **不同平台 → 追加新行**：若旧值无该平台的行（如原本只有 X，现在 Reddit 任务触发），则将新行追加到末尾。
- **同平台 → 覆盖替换该行**：若旧值已有该平台的行（如 X 数据昨天已写入，今天 X 任务再次触发同词），则**定位到旧行并替换**，不累加产生重复行。平台识别以行首 emoji+平台名前缀为锚（如 `🐦 X(Twitter):`）。
- **本地与飞书同步**：每次写入均先更新本地 `keyword_media_heat`，再同步覆盖写入飞书，保证两端一致。
- **无数据则不写该行**，不留空行占位。

#### 同平台替换的伪代码示意

```python
def merge_media_heat(old_text: str, new_line: str) -> str:
    """同平台覆盖、不同平台追加"""
    prefix = new_line.split(':')[0]  # 如 "🐦 X(Twitter)"
    lines = old_text.strip().splitlines() if old_text else []
    replaced = False
    for i, line in enumerate(lines):
        if line.startswith(prefix):
            lines[i] = new_line   # 同平台 → 替换
            replaced = True
            break
    if not replaced:
        lines.append(new_line)    # 新平台 → 追加
    return '\n'.join(lines)
```

### 2.2 本地 state 新增字段

`seo-radar-state.json` 新增两个平级字段

```json
{
  "keywords": ["Veo 3", ...],
  "keyword_records": {"Veo 3": "recXXX001"},
  "keyword_media_heat": {
    "Veo 3": "🐦 X(Twitter): 15条推文 | 最高 12,300❤ | 2,100RT\n更新于 2026-03-27 UTC+8"
  },
  "keyword_media_heat_updated": {
    "Veo 3": "2026-03-27"
  }
}
```

- `keyword_media_heat`：存储完整复合文本（含 `更新于` 行），作为飞书写入的数据源
- `keyword_media_heat_updated`：单独存储最后更新的北京时间日期，便于快速判断数据时效，无需解析全文

### 2.3 `更新于` 行处理规则

`merge_media_heat()` 需特殊处理末尾的 `更新于` 行：**每次合并后无条件替换**，不当作平台行处理：

```python
def merge_media_heat(old_text: str, new_line: str, today_bj: str) -> str:
    prefix = new_line.split(':')[0]
    lines = [l for l in (old_text.strip().splitlines() if old_text else [])
             if not l.startswith('更新于')]   # 先摘採旧的更新时间行
    replaced = False
    for i, line in enumerate(lines):
        if line.startswith(prefix):
            lines[i] = new_line   # 同平台 → 替换
            replaced = True
            break
    if not replaced:
        lines.append(new_line)    # 新平台 → 追加
    lines.append(f'更新于 {today_bj} UTC+8')  # 末尾统一追加最新北京时间
    return '\n'.join(lines)

# today_bj 由脚本执行：
import subprocess
today_bj = subprocess.check_output(['bash','-c','TZ=Asia/Shanghai date +%Y-%m-%d']).decode().strip()
```

### 2.4 本地与飞书的同步策略

**写入顺序**：先更新本地 state，再写飞书。本地 state 是唯一权威来源。

| 场景 | 处理方式 |
|------|----------|
| 飞书写入成功 | 双端一致，正常 |
| 飞书写入失败（网络/超时） | 本地已更新，飞书落后；**下次同词触发时自动修复** |
| 手动编辑飞书字段 | 下次同词触发时本地裆盖飞书编辑，**以本地为准** |

> ⚠️ 禁止手动修改飞书`媒体热度`字段。若需修正数据，直接修改本地 `keyword_media_heat` 后手动运行脚本同步。

### 2.5 支持追加逻辑（UPDATE 模式）

```python
# 如果上面拼接出了 new_line，说明本次有媒体热度数据传入
if new_line:

    new_heat = merge_media_heat(old_heat, new_line, today_bj)

    state.setdefault('keyword_media_heat', {})[args.keyword] = new_heat
    state.setdefault('keyword_media_heat_updated', {})[args.keyword] = today_bj

    existing_record_id = state.get('keyword_records', {}).get(args.keyword)
    if existing_record_id:
        update_record(token, existing_record_id, {'媒体热度': new_heat})
    else:
        fields['媒体热度'] = new_heat
```

## 三、各来源任务的 Prompt 与采集逻辑改造

**【重大细节警惕】** 
当前架构下，数据并非由 Python 上下文直接传递，而是**由大模型（LLM）阅读前置抓取脚本的终端输出（stdout）后，再拼接具体的字符串传给 `write_keyword.py`**。
因此，改造的核心是**修改 cron payload 中的 Prompt 规则**和**底层 bash/python 的打印输出**，让大模型能“看懂并算对”。

### 3.1 seo-twitter（最复杂，需同时改 bash 和 prompt）

**① 改造 Bash 打印格式**（保持原建议，让大模型能看到 RT 数量）：
```python
rts = t.get('public_metrics',{}).get('retweet_count',0)
print(f'[T1][...] [{likes}❤ | {rts}RT] {t["text"][:200]}')
```

**② 改造 Prompt 规则**（向脚本传递离散数字）：
> **媒体热度统计准则（必须严格遵守）**：
> 当你收集某个关键词时，清点包含该词的所有推文。必须由你自己在上下文中计算出：总推文条数（n）、选出带有最高“❤”数量的那一条、并提取出那条推文的 ❤ 和 RT 数量。
> 在最后调用 `write_keyword.py` 时，将这些数值通过参数直接传入（不要自己拼接长字符串）：
> `--x-tweets <总条数> --x-likes <最高❤数> --x-rts <对应的RT数>`

### 3.2 seo-producthunt

**改造 Prompt 规则**：
> 在调用 `write_keyword.py` 写入新词时，从 JSON 中提取该词的 `votes` 数量以及可能存在的排名。传入参数：
> `--ph-votes <得票数> --ph-rank "<如 #1 of the Day>"`

### 3.3 seo-github

**改造 Prompt 规则**：
> 提取对应仓库的 Stars 数，并传入：
> `--gh-stars <Stars数量>`

### 3.4 seo-reddit

**改造 Prompt 规则**：
> 清点含有该词的帖子总数，找出最高 `score`。传入：
> `--reddit-posts <总帖子数> --reddit-score <最高score>`

### 3.5 seo-ainews（新闻源）

**改造 Prompt 规则**：
> 统计该词在几家媒体中出现过。传入：
> `--news-count <总篇数> --news-outlets "<如 TechCrunch · VentureBeat>"`

### 3.6 其他来源（HuggingFace, arXiv, HN, YouTube, 中国站, Platforms）

这些来源无标准化的社交热度指标，**本期不要求输出`媒体热度`字段**，Prompt 保持原样，写入时不用填这段参数。

def update_record(token: str, record_id: str, fields: dict) -> dict:
    url  = f'{FEISHU_HOST}/open-apis/bitable/v1/apps/{APP_TOKEN}/tables/{TABLE_ID}/records/{record_id}'
    body = json.dumps({'fields': fields}).encode()
    req  = urllib.request.Request(
        url, data=body,
        headers={'Content-Type': 'application/json', 'Authorization': f'Bearer {token}'},
        method='PUT'
    )
    with urllib.request.urlopen(req, timeout=15) as resp:
        return json.loads(resp.read())

def merge_media_heat(old_text: str, new_line: str, today_bj: str) -> str:
    """同平台覆盖、不同平台追加"""
    prefix = new_line.split(':')[0]
    lines = [l for l in (old_text.strip().splitlines() if old_text else []) if not l.startswith('更新于')]
    replaced = False
    for i, line in enumerate(lines):
        if line.startswith(prefix):
            lines[i] = new_line
            replaced = True
            break
    if not replaced:
        lines.append(new_line)
    lines.append(f'更新于 {today_bj} UTC+8')
    return '\n'.join(lines)
```

            if res.get('code') != 0:
                raise RuntimeError(str(res))
            out({'ok': True, 'action': 'update', 'record_id': existing_record_id, 'keyword': args.keyword})
            sys.exit(0)
        except Exception as e:
            out({'ok': False, 'error': f'飞书 Update API 失败: {e}'})
            sys.exit(1)

    # ── 新词写入分支（走到这里的都是新词） ──
    seq = next_seq(category_key)
```
