# 迭代方案：发布日期统一定义与高精度取证架构

正如您一针见血指出的，依赖大模型的“先验常识”或“顺便从竞品搜索中抽取”，在严谨的数据清洗场景下是非常**不可靠**的。大模型极易产生幻觉，且顺带抓取极易受到无效摘要的污染。

因此，为了实现**“极度准确”且“成本极低”**的双重目标，我们将彻底抛弃大模型的“心算猜测”，全盘采用**外挂独立验证的微服务架构**。以下是具体的重构方案。

---

## 核心设计一：统一全站「发布日期」的界定边界

在全站的所有抓取脚本（Twitter / YouTube 等 11 个入口）中，严格声明：
**“发布日期”仅指产品向公众开放的最初问世时间。不能在原文中找到就不准填，禁止任何猜测，全部强制留空 (`--release-date ""`) 移交给底层查证中枢！**

---

## 核心设计二：高精度、低成本的独立溯源防线（完全取代原先的大模型估算）

所有留空 `--release-date ""` 的候选词，在进入 `write_keyword.py` 记录前，必须先经过一道我们新建的独立脚本 `verify_date_engine.py` 进行**双重交叉猎采**。

### 第一关：极其可靠的免费结构化接口（非大模型方案）
除了大模型，最低成本且最100%可靠的方式是**“开源数据库直连”**。
对于每一个待验证词，脚本通过轻量级的 REST API 进行探针搜索：
1. **GitHub API（针对开源模型）**：如果该词是模型名，通过接口找 `repository created_at` 字段。
2. **ProductHunt API（针对 SaaS 工具）**：查询 GraphQL 查找该产品的 `launched_at` 时间戳。
3. **Wikipedia API（针对老牌大词）**：查找维基百科右侧资料卡的 `Initial release` 属性。

**优势**：成本绝对 **$0**。只要命中，准确率是 **100%（官方机读数据）**。

### 第二关：专向精准检索 + 廉价强核小模型（兜底防线）
对于第一关接口查不到的词（例如国产小模型、闭源新工具），不可避免地需要用到搜索引擎。此时我们必须发起指向性极强的精准搜索：`"{Keyword} tool release date announcement"`。

**取证控制**：我们**绝不开网页**（太耗Token），只将搜索引擎接口返回的前五条**纯文本摘要片段 (Snippets)** 送给专职的“小模型”审判员。只需向小模型索要不超过 10 个 Token 的硬格式输出 `YYYY-MM-DD` 或 `NULL`。

---

## 核心设计三：执行逻辑——阶梯式级联降级 (Sequential Tiered Fallback)

针对您的提问：这四层是同步跑还是依次降级？在生产级架构中，推荐采用**“依次阶梯式降级”**。

**执行流向图**：
`GitHub API (免费)` ➔ `Wikipedia API (免费)` ➔ `ProductHunt API (免费)` ➔ `GPT-4o-mini (付费兜底)`

### 1. 为什么不推荐同步（并发）验证？
- **事实数据优先权**：前三层返回的是 JSON 格式的 Unix 时间戳或 Wiki 属性，属于 100% 准确的“官方事实”。一旦在 GitHub 或 PH 命中，我们拿到的就是唯一的标准答案。如果此时同步调用 GPT-4o-mini，不仅浪费钱，还可能因为 AI 的语义理解偏差去“质疑”官方事实，反而引入不稳定因素。
- **极端省钱策略（成本领先）**：由于绝大多数知名工具在 GitHub/PH/Wiki 阶段就能免费结案。阶梯式降级确保了主干流量的 **$0 成本**。只有在“免费情报局”全军覆没时，才动用“AI 战法”。

### 2. 只有一种情况建议同步：
如果您对**“速度”**有着极致的变态要求（比如要求全库 100 个词必须在 3 秒内全部洗完），此时可以采用异步并发，但必须设置**优先级权重**（即：如果有 API 返回了日期，强行覆盖大模型的返回结果）。

**结论**：在本项目（SEO雷达）的日级别/小时级别更新节奏下，**“依次降级”是准确性与成本的最优平衡解**。
